# Dynamic AI Insights & Structured Meeting Core Architecture

## Overview
The Pnyx AI Host acts as an active participant during meetings, surfacing real-time intelligence. Currently, the architecture tightly couples the AI's output to a hardcoded Enum of 7 event types (e.g., `decision_candidate`, `agenda_drift`).

While this rigid structure ensures UI stability, it critically bottlenecks the **"Custom AI Host Skills"** feature. If a user defines their AI Host as an "Aggressive Sales Coach," the AI cannot dynamically emit domain-specific insights like `buy_signal` or `competitor_mentioned` because the backend Pydantic models will reject the JSON.

To solve this, we are migrating to a **Structured Core + Extensible Periphery** architecture.

## The Hybrid Architecture

### 1. Structured Core (Rule-Based)
Fundamental meeting mechanics are universal and power core UI tabs (like the "Decisions" and "Discussions" panels). These will remain strictly rule-based and heavily prompted.

**Reserved Core Event Types:**
- `decision_candidate`: Triggered *only* when a firm choice, commitment, or action is agreed upon.
- `open_discussion`: Triggered *only* for unresolved questions, debates, or active conflicts.

### 2. Extensible Periphery (Dynamic AI Insights)
Any event generated by the AI that is *not* a reserved core event will be treated as an **AI Insight**.
Instead of forcing the LLM to map its observations into predefined buckets (like `agenda_drift`), the LLM will be allowed to invent its own `event_type` strings based on the active Persona/Skill.

**Example Custom Event Types:**
- *Engineering Lead Persona:* `tech_debt_warning`, `scope_creep`, `architecture_flaw`
- *Sales Coach Persona:* `buying_signal`, `budget_objection`, `competitor_mention`
- *HR Interviewer Persona:* `red_flag`, `culture_fit_positive`, `evasive_answer`

## Implementation Plan

### 1. Backend Schema Updates
Modify the Pydantic schemas in `backend/app/schemas/ai_participant.py`.
- Remove the strict `HostEventType` Enum validation for the `event_type` field.
- Change `event_type` to a standard `str`.
- (Optional but recommended) Add a `RESERVED_EVENT_TYPES` constant to easily identify core vs dynamic events in business logic.

### 2. Prompt Engineering (`_build_host_prompt`)
Update `backend/app/services/ai_participant.py` to dynamically instruct the LLM on this hybrid approach.

**Proposed Prompt Logic:**
```text
Allowed event_type values:
1. CORE SYSTEM EVENTS:
   - "decision_candidate": ONLY use this when a firm choice or action is agreed upon.
   - "open_discussion": ONLY use this for an unresolved question, conflict, or debate.
2. DYNAMIC INSIGHTS:
   - Because your role is [User's Custom Persona], you MUST generate custom event_type strings for your insights.
   - For example, if you notice a budget concern, output "event_type": "budget_risk". 
   - Ensure your custom event_types are snake_case strings.
```

### 3. Frontend UI Routing (`frontend/src/app/page.tsx`)
Update the React frontend to route incoming WebSocket events dynamically rather than relying on hardcoded Enum checks.

**Proposed Routing Logic:**
```typescript
// Core System Routing
const decisionsForPanel = events.filter(e => e.event_type === 'decision_candidate');
const discussionsForPanel = events.filter(e => e.event_type === 'open_discussion');

// Dynamic Insight Routing (Catch-all)
const insightsForPanel = events.filter(e => 
   e.event_type !== 'decision_candidate' && 
   e.event_type !== 'open_discussion'
);
```
*UI Enhancement:* The UI should render the dynamic `event_type` string as a human-readable badge (e.g., converting `scope_creep` to "Scope Creep") next to the insight content.

## Multi-LLM Provider Configuration
To ensure maximum flexibility and reliability, the AI Host Engine must support multiple foundational models. The system must be configurable via the backend `.env` file to switch between providers without code changes.

**Supported Providers:**
- **OpenAI** (e.g., `gpt-4o`)
- **Anthropic** (e.g., `claude-3-5-sonnet`)
- **Google Gemini** (e.g., `gemini-2.5-flash`)

**Environment Configuration (`.env`):**
```env
# AI Host Provider Configuration
AI_HOST_PROVIDER=anthropic  # Options: openai, anthropic, gemini
AI_HOST_MODEL=claude-3-5-sonnet-20241022

# API Keys (Loaded dynamically based on the active provider)
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
GEMINI_API_KEY=AIza...
```

The backend `AIParticipantEngine` should read `AI_HOST_PROVIDER` on initialization and construct the respective client using the official SDKs or a standardized wrapper (like LiteLLM or PydanticAI if available in the stack).
