# AI Participant Architecture & Dynamic Event System

## Overview
The "AI Host" feature is being completely rebranded and rearchitected as the **"AI Participant"**. 
Rather than just providing rigid "Insights," the AI Participant acts as an active, persona-driven member of the meeting. It surfaces real-time intelligence based strictly on the role it is assigned (e.g., a "Facilitator" will act very differently from a "Team Lead").

Currently, the architecture tightly couples the AI's output to a hardcoded Enum of 7 event types (e.g., `decision_candidate`, `agenda_drift`). This makes the system overly complicated and critically bottlenecks the ability to use Custom AI Skills.

To solve this, we are migrating to a **Structured Core + Extensible Periphery** architecture driven by **Markdown Skill Files**.

---

## 1. Rebuilding the Core (Rule-Based Mechanics)
The current logic for extracting `decision_candidate` and `open_discussion` is overcomplicated and unreliable. To avoid making a tangled system even more complex, **we will rebuild the Decision and Open Discussion extraction from scratch**.

These fundamental meeting mechanics are universal and power core UI tabs (like the "Decisions" and "Discussions" panels). They will remain strictly rule-based and heavily prompted to guarantee UI stability.

**Reserved Core Event Types (Rebuilt from scratch):**
- `decision_candidate`: Triggered *only* when a firm choice, commitment, or action is explicitly agreed upon.
- `open_discussion`: Triggered *only* for unresolved questions, debates, or active conflicts.

---

## 2. Extensible Periphery (Dynamic AI Participant Actions)
Any event generated by the AI that is *not* a reserved core event will be treated as a **Dynamic Participant Action** (formerly "AI Insight"). 

Instead of forcing the LLM to map its observations into predefined buckets (like `agenda_drift`), the LLM will be allowed to invent its own `event_type` strings based on its active **Skill File**.

**Example Custom Event Types by Persona:**
- *Engineering Lead Persona:* `tech_debt_warning`, `scope_creep`, `architecture_flaw`
- *Sales Coach Persona:* `buying_signal`, `budget_objection`, `competitor_mention`
- *HR Interviewer Persona:* `red_flag`, `culture_fit_positive`, `evasive_answer`

---

## 3. The Skill File System (`.md` format)
The behavior, persona, and dynamic event types of the AI Participant are entirely driven by **Markdown Skill Files**. 

### Prebuilt System Skills
We will provide three hardcoded, prebuilt `.md` skill files out-of-the-box. 
- `facilitator.md` **(Default for all users until changed in settings)**
- `advisor.md`
- `chairperson.md`

### Custom Skills via Settings
Users can upload or define their own custom `.md` skill files (e.g., `team_lead.md`, `scrum_master.md`) via the **Settings UI**. Once uploaded, users can select this custom skill for their meetings, and the AI will behave exactly as defined in the file.

### Required Skill File Format
Every `.md` skill file must follow a standardized format to ensure the prompt builder can parse it correctly:

```markdown
---
name: "Team Lead"
description: "An assertive engineering lead focused on unblocking the team and tracking sprint goals."
---

# Role
You are an assertive Engineering Team Lead. Your primary goal is to ensure technical decisions are made quickly, blockers are identified, and scope creep is prevented.

# Goals
1. Identify when technical discussions are going in circles and demand a resolution.
2. Flag any new feature requests as scope creep.
3. Ensure every task has a clear owner.

# Allowed Custom Event Types
You may generate the following custom event_types when you observe relevant behavior:
- `scope_creep`: When a participant suggests work outside the current sprint or agenda.
- `blocker_detected`: When a participant mentions they cannot proceed with a task.
- `tech_debt_warning`: When a sub-optimal technical shortcut is proposed.

# Rules
- Be concise and direct.
- Do not offer general praise; focus on actionable engineering items.
```

## Implementation Plan

### 1. Backend Schema & Prompt Updates
- **Remove Enum:** Modify `backend/app/schemas/ai_participant.py` to remove the strict `HostEventType` Enum. Change `event_type` to a standard `str`.
- **Rebuild Core Prompts:** Rip out the old decision/discussion extraction and replace it with a highly focused, simplified system.
- **Dynamic Prompt Injection:** Update `_build_host_prompt` in `ai_participant.py` to dynamically load the active `.md` skill file (defaulting to `facilitator.md`) and inject its custom event types and rules directly into the LLM context.

### 2. Frontend UI Routing (`frontend/src/app/page.tsx`)
Update the React frontend to route incoming WebSocket events dynamically and rename terminology from "Insights" to "Participant Actions".

**Proposed Routing Logic:**
```typescript
// Core System Routing (Strictly reserved)
const decisionsForPanel = events.filter(e => e.event_type === 'decision_candidate');
const discussionsForPanel = events.filter(e => e.event_type === 'open_discussion');

// Dynamic Participant Actions (Catch-all)
const participantActionsForPanel = events.filter(e => 
   e.event_type !== 'decision_candidate' && 
   e.event_type !== 'open_discussion'
);
```

### 3. Multi-LLM Provider Configuration
To ensure maximum flexibility, the AI Participant Engine must be configurable via the backend `.env` file to switch between providers *without code changes*.

**Supported Providers:**
- **OpenAI** (e.g., `gpt-4o`)
- **Anthropic** (e.g., `claude-3-5-sonnet`)
- **Google Gemini** (e.g., `gemini-2.5-flash`)

**Environment Configuration (`.env`):**
```env
# AI Participant Provider Configuration
AI_PARTICIPANT_PROVIDER=anthropic  # Options: openai, anthropic, gemini
AI_PARTICIPANT_MODEL=claude-3-5-sonnet-20241022

# API Keys (Loaded dynamically based on the active provider)
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
GEMINI_API_KEY=AIza...
```
The backend `AIParticipantEngine` will read `AI_PARTICIPANT_PROVIDER` on initialization and construct the respective client using standard wrappers (e.g., LiteLLM or official SDKs).
